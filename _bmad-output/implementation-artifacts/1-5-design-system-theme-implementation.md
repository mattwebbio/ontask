# Story 1.5: Design System & Theme Implementation

Status: done

## Story

As a user,
I want a visually distinctive app with the Clay colour palette, serif typography, and multiple theme options,
So that On Task feels warm, trustworthy, and personally comfortable from first launch.

## Acceptance Criteria

1. **Given** the Flutter project exists, **When** the design system is implemented, **Then** `lib/core/theme/app_theme.dart` defines all design tokens as named constants: Clay terracotta `#C4623A`, cream `#FDF6EE`, slate `#3D3D3D`, commitment purple, success green, warning amber, and all derived surface, text, and border tokens
2. Four named theme variants exist as `ThemeData` instances: Clay, Slate, Dusk, Monochrome
3. Light and dark variants exist for all four themes (8 total `ThemeData` instances)
4. All themes pass WCAG 2.1 AA contrast: minimum 4.5:1 for body text, 3:1 for large text
5. **Given** the theme is applied, **When** the New York serif font is configured, **Then** the app uses the `.NewYorkFont` platform string on iOS
6. A Playfair Display subset is bundled as a fallback serif for future Android support
7. Font availability is detected via a platform channel before applying New York; the fallback is applied when unavailable
8. **Given** any text widget is rendered, **When** it uses the design system, **Then** no text size is hardcoded — all text styles reference theme text tokens
9. All text scales correctly at all Dynamic Type sizes including the largest accessibility sizes (NFR-A3)

**Dusk — Deep Indigo & Warm Off-White:**
Primary surface: warm off-white `#F8F5F2` (light) / `#12101A` (dark). Accent: deep indigo `#4A3F8C` (light) / `#7B6FCC` (dark). Text: near-black `#1A1820` (light) / `#EDE8F5` (dark). Secondary text: warm muted `#5C5570` (light) / `#9E94C0` (dark). Completion: amber-gold `#C98A2E` (light/dark). Commitment: deepest indigo `#1A1228` (light) / `#EDE8F5` (dark).

**Slate — Charcoal & Gold (UX spec calls this "Forge"):**
Primary surface: near-white `#F5F5F3` (light) / `#1A1A1A` (dark). Accent: warm gold `#C4922A` (light) / `#D4A83A` (dark). Text: dark charcoal `#1A1A1A` (light) / `#F0EFED` (dark). Secondary text: mid-grey `#5C5C5C` (light) / `#9E9E9E` (dark). Commitment: deepest slate `#0D0D0D` (light) / `#F0EFED` (dark).

**Monochrome — Pure Greyscale:**
Primary surface: `#FFFFFF` (light) / `#111111` (dark). Accent: `#333333` (light) / `#DDDDDD` (dark). Text primary: `#111111` (light) / `#EEEEEE` (dark). Text secondary: `#555555` (light) / `#999999` (dark). Commitment: `#000000` (light) / `#FFFFFF` (dark). Completion: `#444444` (light) / `#CCCCCC` (dark).

**WCAG AA verification:** All body text / surface pairs must achieve ≥ 4.5:1. All large text / surface pairs must achieve ≥ 3:1. The contrast test in `test/core/theme/contrast_test.dart` is the enforcement mechanism — it will fail CI if any theme/variant pair falls below standard. Use the WCAG contrast formula: `(L1 + 0.05) / (L2 + 0.05)` where L = relative luminance.

### Type Scale — No Hardcoded Sizes in Widgets

Define named constants in `app_text_styles.dart`. The `TextStyle.fontSize` field on these constants holds the base pt value. Flutter's text engine applies `MediaQuery.textScaler` automatically — widgets never call `textScaler.scale()` manually.

```dart
// app_text_styles.dart — base sizes only; Dynamic Type scaling is automatic
class AppTextStyles {
  // SF Pro (system font — fontFamily: null uses system default)
  static const TextStyle heroTask = TextStyle(
    fontSize: 28, fontWeight: FontWeight.w600, height: 1.2,
  );
  static const TextStyle sectionHeading = TextStyle(
    fontSize: 22, fontWeight: FontWeight.w700, height: 1.2,
  );
  static const TextStyle body = TextStyle(
    fontSize: 17, fontWeight: FontWeight.w400, height: 1.4,
  );
  static const TextStyle secondary = TextStyle(
    fontSize: 15, fontWeight: FontWeight.w400, height: 1.4,
  );
  static const TextStyle caption = TextStyle(
    fontSize: 13, fontWeight: FontWeight.w400, height: 1.4,
  );

  // New York / Playfair Display (fontFamily resolved at runtime by FontConfig)
  // These TextStyles set fontFamily to null here; font is applied via copyWith
  // in AppTheme.buildTheme() using the resolved FontConfig.serifFamily string.
  static const TextStyle voiceCopyPrimary = TextStyle(
    fontSize: 20, fontWeight: FontWeight.w400, height: 1.6,
  );
  static const TextStyle voiceCopySecondary = TextStyle(
    fontSize: 15, fontWeight: FontWeight.w400, fontStyle: FontStyle.italic, height: 1.6,
  );
  static const TextStyle impactMilestone = TextStyle(
    fontSize: 34, fontWeight: FontWeight.w400, height: 1.2,
  );
}
```

**CRITICAL RULE (NFR-A3):** Widgets in feature layers MUST reference `Theme.of(context).textTheme.*` or `AppTextStyles.*` constants — NEVER literal `fontSize: 17` etc. Hardcoded font sizes in widget trees will fail the `app_theme_test.dart` TextStyle inspection test.

### ThemeData Construction Pattern

`app_theme.dart` exposes:

```dart
enum ThemeVariant { clay, slate, dusk, monochrome }

class AppTheme {
  static ThemeData light(ThemeVariant variant, String serifFontFamily) { ... }
  static ThemeData dark(ThemeVariant variant, String serifFontFamily) { ... }
}
```

Each `ThemeData` is built from `AppColors` + `AppTextStyles` + `serifFontFamily`. The `serifFontFamily` string is passed in at construction time from `FontConfig` (resolved at app start via the platform channel).

`MaterialApp.router` in `main.dart` receives `theme:` and `darkTheme:` computed from the active `ThemeVariant` stored in the Riverpod provider. The app currently uses `MaterialApp.router` (set up in Story 1.4) — this story adds `theme:`, `darkTheme:`, and `themeMode:` properties to it. Do NOT switch to `CupertinoApp` — the Cupertino navigation shell comes in Story 1.6.

**Why MaterialApp for now:** Story 1.6 introduces `CupertinoTabScaffold`. The theme tokens established in this story will be consumed by that scaffold. Do not pre-build navigation structure in this story.

### Platform Channel — New York Font Detection

**Channel name:** `com.ontaskhq.ontask/fonts`
**Method:** `isNewYorkAvailable` → returns `bool`

iOS Swift handler in `AppDelegate.swift`:

```swift
// Inside didFinishLaunchingWithOptions, before super call:
let fontChannel = FlutterMethodChannel(
    name: "com.ontaskhq.ontask/fonts",
    binaryMessenger: controller.binaryMessenger
)
fontChannel.setMethodCallHandler { call, result in
    if call.method == "isNewYorkAvailable" {
        let families = UIFont.familyNames
        result(families.contains(".NewYorkFont"))
    } else {
        result(FlutterMethodNotImplemented)
    }
}
```

Flutter `font_channel.dart`:

```dart
const _channel = MethodChannel('com.ontaskhq.ontask/fonts');

Future<bool> isNewYorkAvailable() async {
  if (!Platform.isIOS) return false; // macOS: always false → use SF Pro display
  try {
    return await _channel.invokeMethod<bool>('isNewYorkAvailable') ?? false;
  } on PlatformException {
    return false; // safe fallback
  }
}
```

`FontConfig` class resolved at startup:

```dart
class FontConfig {
  final String serifFamily;
  const FontConfig({required this.serifFamily});
}

@Riverpod(keepAlive: true)
Future<FontConfig> fontConfig(FontConfigRef ref) async {
  final newYorkAvailable = await isNewYorkAvailable();
  return FontConfig(
    serifFamily: newYorkAvailable ? '.NewYorkFont' : 'PlayfairDisplay',
  );
}
```

**macOS note:** New York is available on macOS as well in newer OS versions, but cannot be queried the same way via UIFont. For this story, macOS always uses Playfair Display. A future story can add macOS New York support if needed.

### Theme Provider Pattern

```dart
// lib/core/theme/theme_provider.dart
import 'package:riverpod_annotation/riverpod_annotation.dart';
import 'package:shared_preferences/shared_preferences.dart';
import 'app_theme.dart';

part 'theme_provider.g.dart';

@riverpod
Future<ThemeVariant> themeVariant(ThemeVariantRef ref) async {
  final prefs = await SharedPreferences.getInstance();
  final stored = prefs.getString('theme_variant');
  return ThemeVariant.values.firstWhere(
    (v) => v.name == stored,
    orElse: () => ThemeVariant.clay,
  );
}
```

In `main.dart`, `OnTaskApp.build()` watches both `themeVariantProvider` and `fontConfigProvider`:

```dart
@override
Widget build(BuildContext context, WidgetRef ref) {
  final router = ref.watch(appRouterProvider);
  final variantAsync = ref.watch(themeVariantProvider);
  final fontAsync = ref.watch(fontConfigProvider);

  final variant = variantAsync.valueOrNull ?? ThemeVariant.clay;
  final fontConfig = fontAsync.valueOrNull ?? const FontConfig(serifFamily: 'PlayfairDisplay');

  return MaterialApp.router(
    title: 'OnTask',
    routerConfig: router,
    theme: AppTheme.light(variant, fontConfig.serifFamily),
    darkTheme: AppTheme.dark(variant, fontConfig.serifFamily),
    themeMode: ThemeMode.system,
  );
}
```

`variantAsync.valueOrNull` ensures the app renders immediately with the Clay default while prefs load — no blank screen.

```dart
double contrastRatio(Color fg, Color bg) {
  double lum(Color c) {
    final r = c.red / 255; final g = c.green / 255; final b = c.blue / 255;
    linearize(double v) => v <= 0.04045 ? v / 12.92 : pow((v + 0.055) / 1.055, 2.4).toDouble();
    return 0.2126 * linearize(r) + 0.7152 * linearize(g) + 0.0722 * linearize(b);
  }
  final l1 = lum(fg), l2 = lum(bg);
  return (max(l1, l2) + 0.05) / (min(l1, l2) + 0.05);
}
```

Test body text (17pt) / surface pairs for all 8 theme variants — assert ratio ≥ 4.5. Test large text (22pt+) / surface pairs — assert ratio ≥ 3.0. This test is the WCAG enforcement gate — it fails CI if any theme degrades.

### Riverpod v4 Provider Signature (from Story 1.4 learnings)

Story 1.4 debug log: riverpod_generator v4 changed `*Ref` type names — provider functions use `Ref` from `package:riverpod/riverpod.dart` instead of generated `*Ref` types. If the generated code uses `FontConfigRef` and `ThemeVariantRef`, those ARE still generated by riverpod_generator v4 (the change was for non-annotated providers). Use `@riverpod` annotation and let code generation handle the `Ref` type — do not manually write the provider boilerplate.

### Anti-Patterns — Do Not Do These

- **Do NOT** hardcode `Color(0xFFC4623A)` in any widget — always use the token from `AppTheme` / `Theme.of(context).colorScheme`
- **Do NOT** use `fontSize: 17` in a widget's `TextStyle` — use `AppTextStyles.body` or `Theme.of(context).textTheme.bodyMedium`
- **Do NOT** create a `lib/generated/` folder for build_runner output — generated files co-locate with source (established Story 1.4)
- **Do NOT** switch to `CupertinoApp` in this story — the Cupertino shell comes in Story 1.6
- **Do NOT** add the `Live Activities` platform guard in this story — not relevant until Story 12.1
- **Do NOT** build the Settings UI for theme switching — that is Story 1.10; only the underlying Riverpod provider and `shared_preferences` key are needed here
- **Do NOT** run `build_runner` in CI — local only (confirmed Story 1.2 / ARCH-4)
- **Do NOT** use Material Design 3 colour roles exclusively — `colorScheme` is populated but Cupertino theming will supersede in Story 1.6; set `useMaterial3: true` in `ThemeData` for compatibility but keep all token work in the `AppColors` / `AppTheme` layer
